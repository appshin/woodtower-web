# WoodTower — 웹 테스트판

안드로이드 앱을 만들기 전에 **물리·규칙·밸런스를 폰에서 바로 검증**하기 위한 정적 웹 빌드.
게임 로직은 Kotlin 소스와 1:1 포팅이라, 여기서 튜닝한 상수를 그대로 옮길 수 있습니다.

## GitHub Pages에 올리기 (5분)

```bash
git init
git add .
git commit -m "WoodTower web test build"
git branch -M main
git remote add origin https://github.com/appshin/woodtower-web.git
git push -u origin main
```

GitHub → 저장소 → **Settings → Pages** → Source: `Deploy from a branch`,
Branch: `main` / `/ (root)` → Save.

1~2분 뒤 `https://appshin.github.io/woodtower-web/` 에서 열립니다.
폰 브라우저로 열고 **홈 화면에 추가**하면 전체화면 앱처럼 동작합니다.

> `.nojekyll` 파일이 들어있습니다. 없으면 Jekyll이 일부 경로를 건드릴 수 있습니다.

## 두 가지 모드

**이 기기로 함께하기** — 핫시트. 폰 1대를 돌려 씁니다. 인터넷 불필요.

**다른 기기와 연결하기** — WebRTC(PeerJS). 방장이 4자리 코드를 받고,
나머지가 코드를 입력해 참가합니다. 시그널링만 공용 브로커를 거치고 실제
데이터는 기기 간 직접 전송입니다.

## 안드로이드판과 다른 점 (`WEB-ONLY` 주석 참고)

| | 웹 | 안드로이드 |
|---|---|---|
| P2P | WebRTC / PeerJS 브로커 (**인터넷 필요**) | Nearby Connections (인터넷 불필요) |
| 방 찾기 | 4자리 코드 입력 | 근처 기기 자동 발견 |
| 햅틱 세기 | 조절 불가 (길이로만 표현) | `hasAmplitudeControl()` 시 세기 조절 |
| iOS | **진동 없음** (Safari 미지원) | — |
| 붕괴음 | 노크를 실시간으로 흩뿌림 | 사전 합성한 1.9초 클립 |
| 카메라 | **3D 궤도 (손으로 회전)** | 아직 2단계 시점 전환 — 미포팅 |

**동일한 것**: 타워 모델, 금지 층 규칙, 재적층, 무게중심 안정성, 붕괴 확률 곡선,
마찰/stick-slip 모델, 좌석 기반 재접속 프로토콜, 30초 유예·건너뛰기.

## 붕괴 확률 곡선 (중요)

```
u          = clamp((0.30 - margin) / 0.30, 0, 1)
structural = u^1.7
risk       = structural*0.80 + jerk*structural*0.35 + jerk*0.03 + 0.06*fatigue
```

`margin`은 각 층의 잔여 블록 convex hull 대비 상단 무게중심의 정규화 여유값이고,
`fatigue`는 위로 다시 쌓인 블록 비율입니다.

**0.30 위에서는 거의 평평해야 합니다.** 꽉 찬 층에서 바깥 블록 하나를 빼면
margin이 정확히 0.333으로 떨어지는데, 이건 실제 게임에서 완전히 안전한 수입니다.
초기의 선형 매핑은 이걸 17% 위험으로 취급해서 게임이 3수 만에 끝났습니다.

4000판 시뮬레이션 (사람다운 정책: 바깥 하나만 남기지 않음, 가운데 선호):

| jerk (손떨림) | 평균 | 중앙값 | p10 | p90 | 즉사율 |
|---|---|---|---|---|---|
| 0.1 (조심스럽게) | 25.2수 | 29 | 10 | 34 | 0.5% |
| 0.35 (보통) | 22.3수 | 26 | 6 | 34 | 1.4% |
| 0.9 (잡아챔) | 17.7수 | 17 | 3 | 32 | 2.5% |

실제 젠가 20~40수 범위와 일치하고, 손기술이 유의미하게 반영됩니다.

## 조작 (3D 궤도 카메라 · 탭 후 당기기)

- **블록을 탭** → 선택됨 (호박색으로 밝아지고 외곽선이 생김)
- **선택된 블록을 잡고 바깥으로 당기기** → 뽑기
- **그 외 아무 곳이나 밀기** → 타워 회전 / 시선 높이. 튕기면 관성
- `⟲` → 시점 초기화

제스처가 서로 겹치지 않습니다. 초기 버전은 드래그 첫 몇 px의 방향으로 의도를
추측했는데, 회전하려다 스친 블록이 뽑히는 사고가 났습니다. **먼저 선택하게
하면 추측 자체가 사라집니다.**

빨간 외곽선 = 선택했지만 지금 각도에선 뺄 수 없음 (화면을 돌리세요).

### 픽킹은 면 단위입니다

블록 *중심*의 깊이로 정렬하면 안 됩니다. 카메라를 향해 누운 긴 블록은 중심이
멀어도 끝면이 가장 앞에 나옵니다.

```
yaw=0,  X축 블록(z=+1)   중심 depth 1.746  /  앞면 depth 2.224
        Z축 블록(중앙)   중심 depth 0.965  /  끝면 depth 2.399  ← 실제로 제일 앞
```

그래서 `pickFaces`에 보이는 면을 전부 넣고 면의 깊이로 정렬한 뒤,
가장 앞면부터 point-in-polygon 검사합니다. painter's algorithm과 동일한 순서라
**보이는 것과 잡히는 것이 항상 일치**합니다.

### 손이 닿는 블록 (reach)

블록은 자기 길이 방향으로만 빠집니다. 그 축이 카메라를 향한 성분(`reach`)이
0.26 미만이면 긴 옆면만 보이는 것이므로 잡을 수 없습니다 — 실제로는 테이블을
돌아가야 하는 상황이고, 여기선 화면을 돌리면 됩니다.

| yaw | X축 층 | Z축 층 |
|---|---|---|
| 0° | 잡을 수 없음 | 잡힘 |
| 16°~74° | 잡힘 | 잡힘 |
| 90° | 잡힘 | 잡을 수 없음 |

정면(0°)에서 Z축 층만 잡히는 건 이전 2단계 시점 버전과 정확히 같습니다.
달라진 건 그 사이가 **연속**이라는 점입니다.

렌더링은 정투영 + yaw/pitch, 면 단위 painter's algorithm입니다. 모든 면이
평행사변형으로 투영되므로 캔버스 2D affine 변환으로 나무 텍스처가 정확히
맞아 들어갑니다 (원근 왜곡 없음).

## 폰에서 볼 것

1. 블록을 **천천히** 빼는 것과 **확 잡아채는 것**의 감각 차이 (안드로이드 크롬)
2. 타워를 돌려가며 각 각도에서 어떤 블록이 잡히는지 (0°와 90°에서 확인)
3. 카메라가 거의 정면일 때 — 끝면이 보이는 블록은 **아래로** 당기면 빠집니다
4. 아래층 가운데를 계속 빼도 안정도가 100%를 유지하는지
5. 한 층에 바깥 블록 하나만 남겼을 때 즉시 무너지는지
6. 게스트 폰을 비행기모드로 → 30초 배너 → 해제 → 같은 자리·같은 점수로 복귀

## 게임 횟수

설정 화면 / 방 만들기 화면의 **스테퍼(− 3판 +)와 프리셋 칩(1·3·5·10판)** 으로 정합니다.
범위 1~20판. 온라인에서는 방장이 정한 값으로 전원이 진행합니다.

초기 버전은 range 슬라이더였는데, 어두운 배경에서 트랙이 거의 안 보이고 라벨도
흐린 회색 14px이라 아무도 존재를 알아채지 못했습니다. 결과적으로 모든 판이
조용히 기본값 3판으로 돌아갔습니다.

## 튜닝 포인트

- `collapseChance()` — 위 곡선. 게임이 너무 길거나 짧으면 여기
- `TAP_SLOP` (12px) / `TAP_MS` (500ms) — 탭이 회전으로 오인되면 여기
- `reach < 0.26` — 잡을 수 있는 각도의 문턱
- `slipLengthPx()` — stick-slip 간격. 촉감이 잘거나 뚝뚝 끊기면 여기
- `Sound.updateFriction()` — 40 px/s 무음 문턱. stiction 실감의 핵심
- `REJOIN_GRACE_SECONDS` — 재접속 유예

## 알려진 제약

- PeerJS 공용 브로커(`0.peerjs.com`)는 무료라 가끔 느립니다. 실서비스용 아님
- 방 코드 4자리(31^4 ≈ 92만) — 충돌 시 자동 재시도하지만 동시 접속이 많으면 6자리 권장
- 호스트가 나가면 세션 종료 (호스트 마이그레이션 미구현, 안드로이드판과 동일)
